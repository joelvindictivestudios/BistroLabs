# Bygg en AI-driven restaurangbokningsplattform (Elyra-inspirerad)

## Kontext
Jag vill bygga en AI-driven plattform för restaurangdrift, inspirerad av konkurrenten Elyra Systems (elyrasystems.com), en YC-backad plattform som används av flera svenska restauranggrupper (bl.a. Melanders Group, Götaplatsgruppen). Plattformen ska hantera gästkontakt via telefon och mejl, bygga gästprofiler över tid, och sköta bordsplanering.

**Min befintliga stack (återanvänd där det passar):**
- Next.js, TypeScript, React Native
- AWS: ECS Fargate, ECR, RDS, S3, CloudFront, Cognito
- Supabase, Postgres/pgvector
- Fastify, Prisma, Docker, Strapi
- Tidigare byggt: Tarantula (Electron + OpenAI Whisper + GPT-4o + Linear MCP för mötestranskribering) — samma AI-mönster kan återanvändas här

## Referens: hur konkurrenten är strukturerad (observerat, inte bekräftat)
- Multi-tenant widget: varje restaurang har en egen bokningswidget på en unik slug, t.ex. `widget.elyrasystems.com/{restaurang-slug}` — central config per kund (meny, bordskapacitet, öppettider) snarare än separata deployments
- Partystorlek över en viss gräns eskaleras till manuell hantering via mejl istället för att hanteras automatiskt av widgeten
- Egen AI-telefonagent för bokningar (ej granskad i detalj)

## Produktmoduler att bygga

### 1. Voice Agent (telefonbokning)
- Modell: OpenAI **gpt-realtime** (GA-modell, ersätter gpt-4o-realtime-preview) för produktion, alternativt **gpt-realtime-mini** för lägre kostnad
- Telefoni: Twilio Media Streams (SIP) → WebSocket-bridge (Node/Fastify på ECS) → gpt-realtime
- Function calling mot databasen: `check_availability(date, time, party_size)`, `create_booking(...)`, `get_guest_profile(phone_number)`
- System-prompt per restaurang hämtas dynamiskt (meny, ton, policy) och cachas för att hålla kostnad nere
- Fallback-arkitektur för lägre kostnad: självhostad Whisper (STT) → billig textmodell → billig TTS (Cartesia/OpenVoice) istället för realtime speech-to-speech
- Överväg Pipecat eller LiveKit Agents som ramverk för SIP-bridging och turn-taking istället för att bygga från scratch

### 2. Email Concierge
- Modell: GPT-5.4 (bra kostnad/kvalitet) för svarsgenerering, GPT-5.4 Nano/mini för intent-klassificering
- Pipeline: intent-klassificering → RAG-hämtning (policy/FAQ + tidigare mejlväxling via pgvector) → svarsgenerering i restaurangens ton (few-shot från tidigare mejl) → function calling för att utföra bokning/ändring
- Låg confidence eller klagomål → eskalera till manuell granskning istället för auto-send

### 3. Guest Intelligence (kärnan som binder ihop allt)
- Datamodell (Supabase/Postgres + pgvector):
  ```sql
  guests (id, phone, email, name, created_at)
  guest_interactions (id, guest_id, type, raw_content, summary, embedding vector(1536), created_at)
  guest_profile (guest_id, preferences jsonb, dietary_restrictions, favorite_table, visit_count, avg_spend, last_visit)
  ```
- Efter varje interaktion: sammanfatta med billig modell → embedda med `text-embedding-3-large` (eller `-small` för lägre kostnad) → spara
- Periodisk aggregator (cron på ECS) som slår ihop senaste interaktioner till kompakt `guest_profile`
- Vid ny kontakt: slå upp gäst via telefon/email → hämta profil + topp-N relevanta interaktioner via vektorsökning → injicera som kontext i voice/email-agenten

### 4. Table Allocation
- INTE ett LLM-problem — bygg som klassiskt optimeringsproblem (constraint satisfaction), t.ex. greedy + backtracking eller Google OR-Tools
- LLM kan användas ovanpå för att förklara/föreslå i naturligt språk, inte för själva schemaläggningen

### 5. Campaigns / Analytics
- GPT-5.4-mini räcker för kampanjtext/segmentering baserat på gästdata
- Analytics är standard BI ovanpå Postgres-datan, ingen LLM krävs

## Kostnadsprincip
Realtime-röst är dyrast per minut (~$0.06/min in, $0.24/min ut innan caching) — optimera hårt där (korta cachade prompts, function calling istället för långa svar). Text-delar (mejl, kampanjer) kör billigare modeller och eskalerar bara till starkare modell vid komplexa ärenden.
